Why interleaving matters

Sequential harnesses run benchmark A, then B. By the time B runs, the CPU is hotter, turbo boost has decayed, and background load may have changed. Any measured difference includes these artifacts.

Zenbench runs all benchmarks in shuffled order every round. Both see the same thermal state, same load, same cache pressure. Paired statistics on the round-by-round differences isolate the real performance difference from system noise.

Quick start

[dev-dependencies]
zenbench = "0.1"

[[bench]]
name = "my_bench"
harness = false
use zenbench::prelude::*;

fn bench_sort(suite: &mut Suite) {
    suite.group("sort", |g| {
        g.throughput(Throughput::Elements(1000));
        g.bench("std_sort", |b| {
            b.with_input(|| (0..1000).rev().collect::<Vec<i32>>())
                .run(|mut v| { v.sort(); v })
        });
        g.bench("sort_unstable", |b| {
            b.with_input(|| (0..1000).rev().collect::<Vec<i32>>())
                .run(|mut v| { v.sort_unstable(); v })
        });
    });
}

zenbench::main!(bench_sort);

Features

🔀 Interleaved execution

All benchmarks in a group run in shuffled order each round. Paired statistics detect differences that sequential harnesses miss.

📊 Paired statistics

Bootstrap 95% CI, Wilcoxon signed-rank test, Cohen's d effect size, Spearman drift detection, configurable noise threshold.

🚀 CI regression gates

--save-baseline=main and --baseline=main with exit codes. Auto-update on pass. Cross-run variance inflation.

⏱️ Hardware timing

TSC (rdtsc/rdtscp) on x86_64, cntvct_el0 on aarch64. Auto-calibration. Stack alignment jitter. Overhead compensation.

🔄 Criterion migration

Drop-in compatibility layer. Change one import, zero code changes. Closures can borrow — no 'static needed.

📈 Rich output

Tree and table display modes. JSON, CSV, LLM, Markdown formats. Streaming per-group. Adaptive column layout.
